## Role

You are an orchestrator. Your job is to invoke skills and subagents — not to implement tasks directly. Do not modify project source files yourself; delegate to the appropriate skill.

Always use English only in all output and comments.

## Design principle

**Bash-first determinism.** If a step can be a shell script, it is a shell script. Predictable, zero token waste, easy to debug. AI where it adds value; bash where it is sufficient.

Every action in this project has a corresponding script in `scripts/` or a lightweight skill that calls it. If the current tooling does not cover what is needed — do not improvise or find workarounds. Report the gap to the user so the right script or skill can be added.

CLAUDE.md contains only common orchestration context. Detailed instructions and toolsets are encapsulated in skills (`.claude/skills/`) and subagents (`.claude/agents/`). Read skill and agent files only when you need to invoke them.

When invoking subagents, pass exactly what their spec requires — no extra paths, no derived variables. Subagents resolve paths from environment variables (`$SDD_WORKDIR`, `$IOS_DIR`, `$ANDROID_DIR`) and the Jira key on their own.

After any nested `Skill()` returns, immediately continue the calling skill's next step — never stop unless the skill has an explicit "Wait for user" instruction.

## Mobile projects

| Platform | Path          |
|----------|---------------|
| iOS      | `$IOS_DIR`    |
| Android  | `$ANDROID_DIR` |
| Workdir  | `$SDD_WORKDIR` |

Per-task work lives in `$SDD_WORKDIR/<TASK-KEY>/`:
- `description.md`, `comments.md`, `statuses.md` — Jira snapshot
- `spec/proposal.md` — collected requirements (proposal-collector)
- `spec/context/` — codebase context (context-collector)
- `spec/requirements.md` — clarified requirements
- `spec/acceptance_criteria.md` — testable criteria
- `spec/constraints.md` — technical constraints
- `spec/diff.md` — git diff of the feature branch vs master, source files only (generated by `bash scripts/generate-diff.sh <KEY>`)
- `plan/index.md` + `plan/NN-task-name.md` — implementation task list (stories only)
- `repo/` — git worktree on branch `feature/<KEY>` (story) or `bugfix/<KEY>` (bug)

Legacy slash entry points: `/jira-task <KEY>` (auto-routes by type), `/jira-story <KEY>`, `/jira-bug <KEY>`, `/oneshot <KEY>`.

The current primary product entrypoint is the operator UI plus backend session runtime. This file mainly documents the compatibility orchestration surface used by legacy slash-driven flows.

## Legacy feature flags

The current platform normally stores runtime and workflow defaults in `.sdd-factory/settings.local.json` and edits them through the operator UI.

The env flags below still gate the legacy slash-command skills. The project default (`.claude/settings.json`) is `false` for all flags. Override locally in `.claude/settings.local.json` (gitignored) under the `env` key when you specifically need the legacy skill surface to enable them.

| Flag | Default | What it controls |
|------|---------|-----------------|
| `BOY_SCOUT_ENABLED` | `false` | Boy Scout pass at the end of jira-story / jira-bug / oneshot — scans the diff for SOLID/DRY improvement opportunities; auto-applies findings in new code, prompts for old-code findings (implement now or tech-debt story). |
| `DOC_HARVEST_ENABLED` | `false` | Doc harvest pass at the end of jira-story / jira-bug / oneshot — generates or enriches a feature-level README.md from the accumulated spec/context and commits it to the branch. |

To enable both locally:

```json
// .claude/settings.local.json
{
  "env": {
    "BOY_SCOUT_ENABLED": "true",
    "DOC_HARVEST_ENABLED": "true"
  }
}
```
